# v0.4 migration

Ploidy 0.4.0 adds one primary `debate` tool in front of the existing
12-tool protocol surface. All 13 tools ship in this release; existing
phase-control integrations continue to work.

## Version and transport

Upgrade the package and choose a transport explicitly only when the
default is not appropriate:

```bash
pip install --upgrade ploidy==0.4.0

# Default local MCP transport
python3 -m ploidy  # stdio

# Shared or hosted service
PLOIDY_TRANSPORT=streamable-http python3 -m ploidy
export PLOIDY_URL=http://127.0.0.1:8765
```

For HTTP clients, use `${PLOIDY_URL}/mcp`. `PLOIDY_URL` is a client-side
base URL; the server continues to read `PLOIDY_PORT` and transport
settings.

## Replace one-call legacy tools

### Auto

Before:

```python
await debate_auto(
    prompt="Should we split the ingestion service?",
    context_documents=["Architecture and incident constraints…"],
)
```

After:

```python
await debate(
    prompt="Should we split the ingestion service?",
    mode="auto",
    context_documents=["Architecture and incident constraints…"],
    context_sources=["architecture-review"],
)
```

Both `debate(mode="auto")` and `debate_auto` now require non-empty
`context_documents`. Earlier releases could silently run auto mode with
no Deep-only evidence, making both sides symmetric. 0.4.0 rejects that
invalid service path.

### Solo

Before:

```python
await debate_solo(
    prompt="…",
    deep_position="…",
    fresh_position="…",
)
```

After:

```python
await debate(
    prompt="…",
    mode="solo",
    deep_position="…",
    fresh_position="…",
)
```

Both positions must be non-empty. Ploidy stores and converges the text;
the caller remains responsible for generating the Fresh position in a
clean context.

## Mapping

| 0.3 flow | 0.4.0 entry point |
|---|---|
| `debate_auto(...)` | `debate(mode="auto", ...)` |
| `debate_solo(...)` | `debate(mode="solo", ...)` |
| `debate_auto(..., pause_at="challenge")` | `debate(mode="auto", ..., pause_at="challenge")` |
| `debate_start` → join → positions → challenges → converge | Keep the explicit flow when cross-client isolation or phase control is required |
| status/history/cancel/delete | Keep the existing lifecycle tools |

The 12 legacy tool descriptions are marked deprecated, but 0.4.0 does
not remove them. No removal release is promised here.

## Position barrier and multi-seat change

The explicit protocol no longer discloses another seat's position before
all required positions are submitted. Clients should use the submission
response's barrier/phase fields instead of expecting early transcript
contents.

For `deep_n > 1` or `fresh_n > 1`, each seat owns an independently
generated position and challenge. Do not assume one same-side challenge
is copied to every seat.

## Context provenance and blocking

One-call tools accept:

- `context_sources`: one provenance label per context document.
- `blocked_sources`: case-insensitive terms that must not appear in a
  source label or document body.

The response and persisted configuration include a context manifest with
hashes and sizes. A blocked match fails the call before model inference.

## Authentication migration

The default `bearer` mode preserves `PLOIDY_TOKENS` and the legacy
single `PLOIDY_AUTH_TOKEN` fallback. Bearer-with-no-token is an explicit
unauthenticated local-development configuration, not a public default.

For OAuth protocol interoperability in a controlled environment:

```bash
PLOIDY_AUTH_MODE=oauth
PLOIDY_OAUTH_ISSUER=https://ploidy.example.com
```

For a temporary mixed rollout:

```bash
PLOIDY_AUTH_MODE=both
PLOIDY_OAUTH_ISSUER=https://ploidy.example.com
PLOIDY_TOKENS='{"old-static-token":"tenant-a"}'
```

`both` uses OAuth plus an internal static-token fallback. Tenant
ownership applies consistently to MCP and SSE requests, and missing or
invalid SSE bearer credentials return `401`.

OAuth DCR is auto-approved in 0.4.0 and has no resource-owner login or
consent. It is not public access control, and new client IDs can bypass
per-client rate buckets. Use a gateway/private ingress or controlled
static bearer tokens for admission.

## Container and deployment

Replace floating tags with the immutable release tag:

```text
ghcr.io/heznpc/ploidy:0.4.0
```

The 0.4.0 image defaults to Streamable HTTP and includes the API,
dashboard, metrics, Redis, and CLI runtime extras. Deployment examples
also enable bounded context, rate, and retention defaults.

## Compatibility checklist

- [ ] Client expects 13 tools, or enables only `debate` intentionally.
- [ ] Local MCP configuration relies on the `stdio` default.
- [ ] Shared service configuration sets `PLOIDY_TRANSPORT=streamable-http`.
- [ ] Auto callers always send non-empty `context_documents`.
- [ ] `context_sources`, when present, matches document count.
- [ ] Hosted instances use a gateway/private ingress or controlled static
      bearer credentials; OAuth alone is not treated as admission control.
- [ ] Container references use `0.4.0`, not `latest`.
